# HU-06 - Editar una ubicación existente

## Objetivo
Corregir o completar información de una ubicación ya registrada.

## Dependencias previas
HU-05

## Alcance dentro del MVP
Esta historia debe implementarse sin ampliar el alcance funcional del MVP del cotizador de seguros de daños.

## Reglas de negocio aplicables
- Debe respetarse el flujo y estado definidos por el MVP.

## Criterios de aceptación
- Debe permitir edición parcial.
- Debe recalcular el estado de validación.
- Debe conservar la asociación con la cotización.

## Cobertura objetivo de la HU
- Mínimo 80% sobre la lógica crítica implementada.
- Debe existir trazabilidad entre criterios de aceptación, Gherkin y tests.
- Debe cubrirse al menos un caso feliz y un caso de borde o error controlado cuando aplique.

## Fases TDD esperadas
### Red
- Diseñar tests primero.
- Mapear cada escenario Gherkin a pruebas.
- Confirmar fallo inicial.

### Green
- Implementar el mínimo código para pasar los tests.
- No agregar comportamiento fuera de la HU.

### Refactor
- Mejorar diseño interno sin alterar comportamiento observable.
- Mantener pruebas en verde.

## Gherkin
```gherkin
Feature: Edición de ubicación

  Scenario: Completar una ubicación previamente incompleta
    Given que existe una ubicación en estado "INCOMPLETA"
    When el usuario registra los datos faltantes
    Then el sistema actualiza la ubicación
    And recalcula su estado de validación
    And la marca como "COMPLETA"

  Scenario: Editar parcialmente una ubicación
    Given que existe una ubicación registrada
    When el usuario modifica solo parte de sus datos
    Then el sistema actualiza únicamente los campos enviados

```
