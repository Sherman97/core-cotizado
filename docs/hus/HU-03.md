# HU-03 - Guardar datos generales de la cotización

## Objetivo
Registrar datos generales sin sobrescribir información no enviada.

## Dependencias previas
HU-01

## Alcance dentro del MVP
Esta historia debe implementarse sin ampliar el alcance funcional del MVP del cotizador de seguros de daños.

## Reglas de negocio aplicables
- La actualización de datos generales debe ser parcial.

## Criterios de aceptación
- Debe permitir actualización parcial.
- Debe guardar cliente, producto, moneda, observaciones y demás campos definidos para el MVP.
- Debe actualizar fecha de modificación.
- No debe sobrescribir información no enviada.

## Cobertura objetivo de la HU
- Mínimo 80% sobre la lógica crítica implementada.
- Debe existir trazabilidad entre criterios de aceptación, Gherkin y tests.
- Debe cubrirse al menos un caso feliz y un caso de borde o error controlado cuando aplique.

## Fases TDD esperadas
### Red
- Diseñar tests primero.
- Mapear cada escenario Gherkin a pruebas.
- Confirmar fallo inicial.

### Green
- Implementar el mínimo código para pasar los tests.
- No agregar comportamiento fuera de la HU.

### Refactor
- Mejorar diseño interno sin alterar comportamiento observable.
- Mantener pruebas en verde.

## Gherkin
```gherkin
Feature: Registro de datos generales

  Scenario: Guardar datos generales correctamente
    Given que existe una cotización en estado "DRAFT"
    When el usuario registra los datos generales válidos
    Then el sistema guarda la información general de la cotización
    And actualiza la fecha de modificación

  Scenario: Actualización parcial de datos generales
    Given que la cotización ya tiene información general registrada
    When el usuario actualiza solo algunos campos
    Then el sistema modifica únicamente los campos enviados
    And conserva los campos no enviados

```
