# HU-11 - Excluir ubicaciones inválidas del cálculo

## Objetivo
Evitar que una ubicación sin datos mínimos entre al cálculo.

## Dependencias previas
HU-10

## Alcance dentro del MVP
Esta historia debe implementarse sin ampliar el alcance funcional del MVP del cotizador de seguros de daños.

## Reglas de negocio aplicables
- Las ubicaciones inválidas no deben entrar al cálculo.

## Criterios de aceptación
- Una ubicación sin datos mínimos no debe calcularse.
- Debe quedar registrada una alerta o razón.
- El proceso no debe fallar si hay otras ubicaciones válidas.

## Cobertura objetivo de la HU
- Mínimo 80% sobre la lógica crítica implementada.
- Debe existir trazabilidad entre criterios de aceptación, Gherkin y tests.
- Debe cubrirse al menos un caso feliz y un caso de borde o error controlado cuando aplique.

## Fases TDD esperadas
### Red
- Diseñar tests primero.
- Mapear cada escenario Gherkin a pruebas.
- Confirmar fallo inicial.

### Green
- Implementar el mínimo código para pasar los tests.
- No agregar comportamiento fuera de la HU.

### Refactor
- Mejorar diseño interno sin alterar comportamiento observable.
- Mantener pruebas en verde.

## Gherkin
```gherkin
Feature: Exclusión de ubicaciones inválidas

  Scenario: Excluir ubicación inválida
    Given que una cotización tiene una ubicación inválida
    And tiene al menos una ubicación válida
    When el usuario ejecuta el cálculo
    Then la ubicación inválida no debe incluirse en el cálculo
    And el sistema debe registrar la razón de exclusión
    And debe calcular las demás ubicaciones válidas

```
